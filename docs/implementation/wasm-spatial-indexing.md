# Prompt 33 - WASM Spatial Indexing

## Overview

The Prompt 33 spatial indexing stack is implemented in [src/engine/wasm/SpatialIndexWASM.ts](../../src/engine/wasm/SpatialIndexWASM.ts). It exposes a stable query API for bounding-box intersection and nearest-neighbor lookups while allowing the runtime to resolve either:

- A WebAssembly-backed leaf scanner for the hot numeric loops.
- A pure JavaScript fallback that preserves the same result contract.

The current implementation uses a kd-partitioned leaf layout built in TypeScript and a compact WebAssembly kernel for range scanning and distance evaluation. This satisfies the prompt requirement for a performant WASM-backed spatial query bridge without forcing downstream consumers to know whether the backend is WASM or JavaScript.

## Build And Packaging Constraints

The WASM kernel source lives in [src/engine/wasm/spatialIndexKernel.wat](../../src/engine/wasm/spatialIndexKernel.wat).

The generated byte payload lives in [src/engine/wasm/generated/spatialIndexKernel.ts](../../src/engine/wasm/generated/spatialIndexKernel.ts).

Generation command:

```bash
npm run wasm:build:spatial-index
```

Generation pipeline notes:

- `wabt` is a dev dependency only. It is not shipped to the browser.
- The browser consumes a generated `Uint8Array` module, not a fetched `.wasm` URL.
- This avoids asset-path drift across Vite dev, preview, and build environments.
- If the WAT kernel changes, the generated TypeScript payload must be regenerated before commit.

Runtime constraints:

- The kernel currently targets 2D bbox and nearest-neighbor prefiltering only.
- The exported WASM memory is fixed-size and intentionally small because the kernel is a narrow accelerator, not a full geometry runtime.
- If WebAssembly instantiation fails, the engine falls back to JavaScript and exposes the fallback reason through the runtime status surface.

## Consumer Contract

Consumers should depend on the exported API from [src/engine/wasm/index.ts](../../src/engine/wasm/index.ts):

- `SpatialIndexWASM.create(records, options)`
- `queryBoundingBox(query)`
- `queryNearest(query)`
- `benchmark(querySuite)`
- `getCapability()`

This contract stays stable regardless of whether the resolved backend is `wasm` or `javascript`.

## Benchmark Guidance

The benchmark surface compares the active backend against a deterministic JavaScript brute-force baseline. It is intended to answer two questions:

- Is the index reducing candidate scans relative to baseline?
- Is the WASM path materially improving the hot numeric portion of the workload?

The benchmark UI in Toolbox uses synthetic but deterministic study-area records derived from the active project bounding box. This keeps measurements reproducible until later prompts wire real parcel, POI, or raster consumers directly into the same engine.
